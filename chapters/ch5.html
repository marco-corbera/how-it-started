Chapter 05 — 2004 to 2010

Web 2.0

Someone had to name the shift. Tim O'Reilly did it in 2004 with the term "Web 2.0." It wasn't a new version of the web's software — it was a new idea of what the web was for. Not a publishing medium. A participation medium. Not something you read. Something you used.

Ajax: The Technique That Changed Everything

Ajax — Asynchronous JavaScript and XML — was not a new technology. It was a pattern: use JavaScript to fetch data in the background, then update just the part of the page that changed. No full reload required.

Gmail launched in 2004. It felt nothing like a web page — it felt like a desktop app. You could read emails, compose, search, all without a single page reload. Google Maps launched in 2005 and you could drag the map and watch new tiles flow in seamlessly.

The demonstration effect was immediate. Within two years, every major web application was using Ajax. The user experience of the web changed fundamentally. 

// The humble XMLHttpRequest that started it all
var xhr = new XMLHttpRequest();
xhr.open('GET', '/api/data', true);
xhr.onreadystatechange = function() {
  if (xhr.readyState === 4 && xhr.status === 200) {
    var data = JSON.parse(xhr.responseText);
    document.getElementById('result').innerHTML = data.message;
  }
};
xhr.send();

jQuery: Fix the Web, Please

In 2006, John Resig released jQuery with a simple promise: "write less, do more." It solved a real problem — browsers were wildly inconsistent in how they implemented JavaScript APIs. What worked in Firefox didn't work in IE. What worked in Safari was different again.

jQuery abstracted over all of it. One library that handled the differences so you didn't have to. It also gave JavaScript a much cleaner syntax for DOM manipulation. 

// Before jQuery: painful
document.getElementById('box').style.display = 'none';

// After jQuery: readable
$('#box').hide();

At its peak, jQuery was used on over 75% of all websites. It's still used on roughly 70% of websites today. It succeeded so completely that it eventually trained a generation of developers who couldn't imagine the web without it — and that dependency eventually became a problem of its own.

Historical Context

Why jQuery Exists — and Why It Eventually Became Less Necessary

jQuery existed because browsers disagreed on how to implement the same standards. As browsers matured and standardized, the problems jQuery solved became less severe. By 2015, modern browsers were consistent enough that many tasks jQuery handled could be done in plain JavaScript. The need for the library declined. But by then, half the internet was already built on it.

The Rise of Social Media

Web 2.0 was also the era when the web became about people connecting with people, not just people browsing content.

2003: MySpace launches. Users can customize their profiles with CSS (sort of), add songs, and connect with friends. It becomes the most visited site in the U.S. by 2006.

2004: Facebook launches from a Harvard dorm. Clean. Simple. Real names. No glittery backgrounds. By 2008 it has 100 million users.

2005: YouTube launches. User-generated video at scale. Acquired by Google in 2006 for $1.65 billion — considered insane at the time.

2006: Twitter launches. 140-character posts. Real-time public conversation. Nobody understood what it was for, then everyone became addicted to it.

The shift was profound: ordinary people became publishers. The web was no longer written by a small technical class and read by everyone else. Everyone could write now.

JSON Quietly Replaces XML

Ajax originally used XML to transfer data between browser and server. XML is verbose, hierarchical, and a pain to parse in JavaScript. JSON (JavaScript Object Notation) was a simpler alternative: data as JavaScript objects, which the browser could read natively without parsing. 

// XML (tedious)
<user>
  <name>Marco</name>
  <role>developer</role>
</user>

// JSON (clean)
{ "name": "Marco", "role": "developer" }

JSON won. By 2010, almost all web APIs used JSON. The REST architectural style — a set of conventions for designing web APIs using standard HTTP methods — became the default way to expose data on the web.

What Web 2.0 Got Wrong

The participation web had a dark side that wasn't obvious at the time. If users generate content and connections are the product, then the incentive is to maximize engagement — keep people on the platform as long as possible.

Outrage keeps people engaged longer than happiness. Controversy generates more replies than agreement. These dynamics weren't designed intentionally — they emerged from the optimization for engagement. By the time anyone noticed the pattern, the systems were embedded in the infrastructure of daily communication.

Web 2.0 also centralized a web that had been designed to be decentralized. The original web had no owners. Every site was independent. Web 2.0 funneled attention into a handful of platforms that controlled the distribution of everything.

"The web was supposed to be a place where anyone could say anything to anyone. Instead, it became a place where a few companies decided what everyone could say."
