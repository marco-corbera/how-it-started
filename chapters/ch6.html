Chapter 06 — 2010 to Now

The Modern Web

The web was built for documents. Then it became an application platform. Then it became the operating system. Today, the line between a "website" and a "native application" is mostly a question of distribution. Understanding the modern web means understanding how we got here, and where the tension points still are.

HTML5: Not Just New Tags

HTML5 arrived officially in 2014, though browsers had been implementing pieces of it for years. The name was misleading — HTML5 wasn't just a new version of HTML. It was a platform: a collection of APIs and capabilities that turned the browser into a runtime environment for applications.

  • Canvas API — draw graphics programmatically, render 2D and 3D
  • Web Storage — localStorage and sessionStorage, persistent data without cookies
  • WebSockets — real-time bidirectional communication
  • Geolocation — access device location (with permission)
  • Native video and audio — no plugin required, Flash's death sentence
  • Web Workers — background threads in JavaScript
  • Service Workers — the foundation of Progressive Web Apps

These weren't cosmetic changes. They were the infrastructure for an entirely different kind of web application.

Mobile Changes Everything

The iPhone launched in 2007. Android in 2008. By 2013, mobile traffic had overtaken desktop on many major sites. The web had been designed for big monitors and mouse input. Suddenly it had to work on a 3-inch screen you controlled with your thumb.

Ethan Marcotte coined the term Responsive Web Design in 2010. The idea: one codebase, one design system, that adapts to any screen size. CSS media queries let you write rules that applied only at certain viewport widths. 

/* Desktop first */
.container { width: 1200px; }

/* Adapt to smaller screens */
@media (max-width: 768px) {
  .container { width: 100%; padding: 0 16px; }
}

Responsive design became the default approach within a few years. Mobile-first — designing for small screens first, then scaling up — became the dominant methodology by 2015.

The JavaScript Framework Era

As web applications became more complex, the traditional model (jQuery + server-rendered HTML) started showing its limits. Managing state across a large application, keeping the UI in sync with data, handling complex user interactions — these were hard to do cleanly with the tools of 2008.

A new class of tools emerged: JavaScript frameworks that took over rendering entirely.

  • Backbone.js (2010) — first major MVC framework for the frontend
  • AngularJS (2010) — Google's framework, two-way data binding
  • React (2013) — Facebook's library, component model, virtual DOM
  • Vue (2014) — lighter, more approachable alternative to Angular
  • Angular 2+ (2016) — complete rewrite of AngularJS
  • Svelte (2016) — compiles components at build time, no runtime

React's component model became the dominant mental model for UI development. The idea: UI is a function of state. Change the state, the UI updates automatically. This reduced an enormous class of bugs that came from manually keeping the DOM in sync.

Core Concept

Why Frameworks Exist

Frameworks are responses to complexity. When you have a few interactive elements on a page, jQuery is fine. When you have a complex application with hundreds of components, shared state, routing, and async data — you need structure.

The cost of frameworks is real: bundle size, build tooling, learning curve, and a pace of change that makes today's best practice tomorrow's antipattern. The web is still figuring out the right tradeoffs.

Node.js: JavaScript on the Server

In 2009, Ryan Dahl released Node.js — a runtime that let JavaScript run outside the browser, on the server. The motivation was clear: why learn two languages (JavaScript for frontend, Python/Ruby/PHP for backend) when you could use one?

Node's non-blocking I/O model made it particularly good at handling many concurrent connections — useful for real-time applications like chat, notifications, and live updates.

The npm ecosystem that grew around Node became the largest package registry in the world. Today, node_modules folders are the subject of infinite jokes about their size. But npm also democratized code sharing at a scale that wasn't possible before.

HTTPS Everywhere

In the early web, HTTPS (HTTP over TLS/SSL) was used only for sensitive transactions — logins, payments. Everything else was plain HTTP: unencrypted, readable by anyone between you and the server.

In 2014, Google announced they'd give a ranking boost to HTTPS sites. In 2015, Let's Encrypt launched — free, automated SSL certificates for anyone. In 2018, Chrome started marking all HTTP sites as "Not Secure."

Today, HTTPS is the default. The web is encrypted end-to-end for the first time in its history.

Where We Are Now

The web in 2024 is the most capable development platform ever built. You can build photo editors, video games, collaborative documents, 3D visualizations, real-time communication tools — all running in a browser tab, on any device.

But capability comes with complexity. The average webpage today sends more data than the entire ARPANET did in its first year of operation. Build tools, bundlers, transpilers, type systems, testing frameworks, deployment pipelines — the infrastructure around building a web app has become its own specialty.

The counter-movement exists too: HTMX, Astro, 11ty, server components — approaches that question whether all this JavaScript complexity was actually necessary. The pendulum swings back toward simplicity. It always does.

Full Circle

HTMX and the Return to Hypermedia

HTMX is a library that extends HTML with attributes to make any element capable of making HTTP requests and updating parts of the page. No JavaScript required for most interactions.

It's a philosophical statement: the web's original model — hypermedia, HTTP, stateless requests — was right. We overcomplicated it. The complexity was the accident, not the requirement.

This site is built with HTMX. That's intentional.

"The web is not finished. It was never supposed to be. Tim Berners-Lee designed a system that could evolve. The fact that we're still arguing about the right way to build for it means it's working exactly as intended."
